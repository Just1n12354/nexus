## 1. Was Ben ist

Ben ist eine lokale Instanz von **Hermes Agent** (v0.18.2, upstream `7b5ba20`) auf dem
GX10. Zwei Schichten, die man nicht verwechseln darf:

- **Hermes = die Hülle:** Agent-Loop, Tools, Gateway, Sessions, Memory, Skills, MCP.
- **Qwen3.6 = das Gehirn:** Sprache, Planung, Tool-Auswahl.

Ben antwortet nicht nur, er **handelt**: Terminal, Dateien, Code, Prozesse, Git, Bilder,
Websuche, Skills, MCP. Genau darin liegt auch sein Risiko — ein Modellfehler wird zu einer
falschen *Systemaktion*, nicht nur zu falschem Text.

## 5. Schwächen — die Vertrauensgrenze

1. **Plausibles Auffüllen (Kernproblem).** Das Modell erfindet Details, wenn ihm Kontext
   fehlt, statt „ich weiss es nicht" zu sagen. Das ist auch die Wurzel der
   Memory-Halluzination beim Minecraft-Bot.
2. **Falsche Selbsterklärungen.** Ben beschreibt seine eigene Konfiguration teilweise
   falsch. Was er über sich sagt, ist kein Beweis — nachprüfen.
3. **Selbst-Update ist gefährlich.** Ben darf seine eigene Installation nicht ohne Aufsicht
   aktualisieren.
4. **Hängende Streams** kamen vor; Timeout-Netz (900 s) fängt sie ab.
5. **Modellgrenze.** 35B-A3B ist ein MoE mit 3B aktiven Parametern — schnell und günstig,
   aber kein Opus. Bei langen Ketten bricht die Präzision ein.

**Vertrauensmodell:** Ben ausführen lassen, aber Ergebnisse verifizieren. Kein
unbeaufsichtigtes Schreiben in produktive Systeme.

## 7. NEU (13.07.): Remote-Nutzung von anderen Geräten

Ben lief bisher nur *auf* dem GX10. Ab jetzt kann ein zweites Gerät (Raspberry Pi, auch
ausserhalb des Heimnetzes) über **Tailscale** eine eigene Hermes-Installation fahren, die
das GX10-Hirn mitbenutzt:

```
Raspberry Pi (justpi)  ──Tailscale──▶  gx10.tail8b5081.ts.net:8000  ──▶  Qwen3.6
   eigener Hermes-Client                 vLLM des GX10
```

Das Setup-Paket dafür liegt auf dem NAS:
**`/mnt/justnas/Dokumente/Hermesconfig/`** — `README.md`, `setup-remote-hermes.sh`,
`config.remote.yaml`, `connection.env` (enthält den Key), `SOUL.md`.

Wichtig dabei:

- Auf dem Pi läuft nur die **Hülle** (Hermes-Client, leichtgewichtig). Das Modell bleibt
  auf dem GX10 — der Pi rechnet nichts.
- Der GX10 muss laufen, sonst hat der Pi kein Hirn.
- Der Pi bekommt eine **eigene** `.hermes`-Umgebung: eigene Sessions, eigenes Memory,
  eigene Skills. Es ist *nicht* derselbe Ben — es ist ein zweiter Hermes am selben Hirn.

### Auth-Proxy: war kaputt, am 13.07. repariert

Der Zugang läuft über einen **Caddy-Auth-Proxy** (Docker-Container `vllm-auth-proxy`,
`caddy:2`, `--network host`, `restart: unless-stopped`):

```
Remote-Client ──Bearer-Key──▶ gx10:8001 (Caddy)  ──▶  127.0.0.1:8000 (vLLM)
                              kein/falscher Key → 401
```

- **Config:** `/home/justin/services/vllm-auth-proxy/Caddyfile`
- Lauscht **nur** auf den Tailscale-IPs (`100.75.47.118`, `fd7a:115c:a1e0::3239:2f77`),
  Port 8001 — nicht auf localhost, nicht im LAN.
- `flush_interval -1` → Streaming (SSE) wird nicht gepuffert.

**Was passiert war:** Beim Docker-Umbau am 12.07. gegen 23:00 (vLLM zog in den Container
`vllm-qwen36`) verschwand die Datei `Caddyfile`. Docker legte beim Neustart pflichtschuldig
ein leeres *Verzeichnis* dieses Namens an, Caddy fand statt seiner Config einen Ordner und
starb mit **Exit 127**. Der Proxy lag seither tot da, während vLLM auf `0.0.0.0:8000`
keylos erreichbar war.

**Fix am 13.07.:** Caddyfile neu geschrieben, Verzeichnis entfernt, Container gestartet.
Verifiziert: ohne Key → 401, falscher Key → 401, richtiger Key → 200 und das Modell
antwortet.

### Restrisiko (offen)

vLLM selbst published weiterhin `0.0.0.0:8000` **ohne Key**. Wer im Tailnet oder im lokalen
LAN ist, kann Port 8000 direkt ansprechen und den Proxy umgehen. Aus dem Internet nicht
erreichbar (kein Portforward, `ufw` inaktiv).

Für Justins **eigene** Geräte ist das unkritisch. Bevor ein **fremdes** Gerät ins Tailnet
kommt, muss das Portmapping des Containers auf `127.0.0.1:8000:8000` geändert werden —
sonst ist der Key umgehbar.
